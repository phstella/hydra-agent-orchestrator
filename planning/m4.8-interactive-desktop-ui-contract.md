# M4.8 Interactive Desktop UI Contract

Last updated: 2026-02-25

## 1. Purpose

Define the desktop-first UI contract for `M4.8` interactive orchestration so implementation remains aligned with the orchestration console mock direction.

## 2. Target Viewports

1. Primary: `>= 1280px` width.
2. Minimum in scope: `>= 1024px` width.
3. Out of scope for this milestone: `< 1024px` optimization and touch-first UX.

## 3. Layout Contract

At `>= 1280px`:
1. Create panel is always visible.
2. Focused terminal panel is dominant center surface.
3. Running-lanes rail is always visible.
4. Lane detail context is visible above terminal panel.

At `1024px - 1279px`:
1. Create panel remains available in compact form.
2. Running-lanes rail remains visible with reduced card density.
3. Focused terminal and intervention controls remain continuously available.

## 4. Information Architecture Contract

The interactive orchestration route must expose in one surface:
1. create-session controls (adapter, prompt, safety toggles, launch)
2. running-lanes rail with lifecycle and lane identity
3. focused terminal output for selected lane
4. per-lane intervention controls (send input, stop, resize if present)
5. lane-local error and status context

## 5. Lane Card Contract

Each lane card should show:
1. adapter key
2. lane identity hint (session-id suffix or lane alias)
3. lifecycle state (`running`, `completed`, `failed`, `stopped`)
4. minimal activity signal (event count and/or elapsed indicator)
5. selected-lane visual state

Duplicate adapter lanes must be distinguishable without opening details.

## 6. Focused Terminal Contract

1. Terminal header must show selected lane identity and adapter key.
2. Terminal stream source must switch with lane selection within one polling cycle.
3. Bounded buffering and stable auto-scroll behavior must remain intact.
4. Terminal end-state label must be explicit (`completed`, `failed`, `stopped`).

## 7. Intervention Contract

1. Input composer binds only to selected lane while lane is running.
2. Stop action binds only to selected lane while lane is running.
3. Disabled states and validation errors must be explicit.
4. Action feedback must be lane-local and must not overwrite sibling lane status.

## 8. Error Surface Contract

For launch, polling, or intervention failures:
1. show compact error status on affected lane card
2. show contextual error in focused panel when that lane is selected
3. include one actionable recovery hint

No silent failures and no UI dead-ends requiring app restart for transient errors.

## 9. Design System Contract

1. New UI uses existing tokenized components and CSS variables.
2. No hardcoded hex values in M4.8 UI changes.
3. Typography, spacing, and color usage follow existing design-system primitives.

## 10. Validation Contract

Minimum required smoke checks:
1. orchestration interactive shell renders create panel + focused terminal + lanes rail
2. two lanes with same adapter can be created
3. lane selection changes focused terminal source
4. input send is lane-isolated
5. stop action is lane-isolated
6. lane-local polling error does not collapse sibling lane

Required command set:
1. `cargo test --workspace --locked --offline`
2. `cargo clippy --workspace --all-targets --locked --offline -- -D warnings`
3. `npm run lint` (frontend)
4. `npm run test:smoke` (frontend)
