# M4.8 Local Execution Pack (Interactive Orchestration Console, Mockup-Aligned)

Last updated: 2026-02-25

## 1. Purpose

This is the canonical local execution tracker for `M4.8`.

`M4.8` closes the orchestration gap between current interactive capability and the target operator flow shown in the orchestration console mock:
1. one surface to create sessions, monitor active lanes, and intervene per lane
2. explicit support for multiple simultaneous sessions of the same adapter type
3. session-lane identity as first-class state (`session_id` scoped operations)

This milestone is explicitly separate from race/scoring behavior.

## 2. Locked Decisions

1. Interactive orchestration and race orchestration are separate feature tracks.
2. Interactive lane identity is session-based (`session_id`), not adapter-key-based.
3. Duplicate adapter sessions are first-class (for example two `codex` lanes at once).
4. Race IPC, scoring, review, and merge semantics are out of scope and must not change.
5. Collaboration workflows remain Phase 5 scope and are out of scope for this milestone.
6. Delivery is desktop-first (`>=1280px` primary, `>=1024px` minimum).
7. Existing design-token policy remains non-negotiable (no hardcoded hex colors).

## 3. Completion Gate

`M4.8` is complete only when all of the following are true:
1. Operator can spawn/manage multiple concurrent interactive sessions from one orchestration surface.
2. Operator can launch multiple sessions of the same adapter type in one run window.
3. Session selection, terminal focus, polling cursor, input writes, and stop actions are lane-scoped.
4. Per-session artifacts persist under unique session IDs and remain replayable.
5. Failure handling is isolated by lane (one lane failure does not corrupt sibling lanes).
6. Existing race behavior is regression-free.
7. Required backend + frontend validation matrix passes.

## 5. Local Sub-Milestones

### M4.8.1 Orchestration Shell and IA Convergence

- Labels: `phase-4`, `area-ui`, `type-feature`
- Estimate: `M`
- Dependencies: `M4.7`
- Problem: Current interactive flow is functionally complete but still tab/session-list centric, not orchestration-console centric.
- Scope:
1. Introduce an orchestration-focused interactive surface aligned with mockup structure:
   create panel, focused live output, running-lanes rail.
2. Keep legacy interactive route reachable for fallback while making orchestration path primary.
3. Preserve existing safety/adapter gating behavior in the new shell.
- Acceptance Criteria:
1. Operator can execute create -> monitor -> intervene loop from one interactive surface.
2. New shell remains usable at `>=1024px`.
3. Existing interactive functionality remains accessible (no feature loss).
- Out of Scope: race cockpit redesign and Phase 5 collaboration surfaces.

### M4.8.2 Session-Lane Identity Model Hardening

- Labels: `phase-4`, `area-ui`, `area-core`, `type-feature`
- Estimate: `M`
- Dependencies: `M4.2`, `M4.6`
- Problem: Some interaction patterns still visually bias to adapter identity; duplicate adapter lanes require strict session identity boundaries.
- Scope:
1. Ensure all interactive client maps are keyed by `session_id`:
   events, poll cursors, poll errors, selected lane, action states.
2. Ensure backend command handlers remain strictly `session_id` addressed for control paths.
3. Add explicit lane labels in UI to avoid ambiguity when adapter keys repeat.
- Acceptance Criteria:
1. Two or more lanes with same adapter key are distinguishable and controllable.
2. No control path uses adapter key as authority for interactive actions.
3. State transitions remain stable during rapid selection changes.
- Out of Scope: global lane naming presets and collaboration role metadata.

### M4.8.3 Multi-Instance Session Creation Flow

- Labels: `phase-4`, `area-ui`, `area-core`, `type-feature`
- Estimate: `M`
- Dependencies: `M4.8.1`, `M4.8.2`
- Problem: Operator must be able to spawn N concurrent sessions of one adapter without implicit dedupe behavior.
- Scope:
1. Verify and enforce create flow supports repeated launches for same adapter.
2. Improve create UX with clear lane initialization feedback and error handling.
3. Keep experimental/safety gating unchanged while allowing duplicate adapter launches.
- Acceptance Criteria:
1. Launching `codex` or `claude` multiple times yields distinct active lanes.
2. Session creation errors are lane-local and actionable.
3. Safety gates still block invalid launches with clear reason text.
- Out of Scope: bulk launch presets and workflow templates.

### M4.8.4 Running-Lanes Rail and Selection Contract

- Labels: `phase-4`, `area-ui`, `type-feature`
- Estimate: `M`

   session alias/id snippet, adapter key, lifecycle, event activity, elapsed time.

3. Preserve per-lane stop affordance for running lanes.
- Acceptance Criteria:
1. Lane list can display repeated adapter keys without ambiguity.
2. Selected lane is visually explicit and drives center focus.
3. Running vs terminal lane states are immediately visible.
- Out of Scope: score/ranking fields (race-only concern).

### M4.8.5 Focused Terminal and Stream Isolation

- Labels: `phase-4`, `area-ui`, `area-core`, `type-feature`
- Estimate: `M`
- Dependencies: `M4.8.4`
- Problem: Orchestration flow requires terminal focus to switch quickly while preserving per-lane stream continuity.
- Scope:
1. Bind focused terminal strictly to selected lane session.
2. Keep bounded buffering and poll cursor continuity per lane.
3. Surface lane-scoped polling failures without impacting sibling lanes.
- Acceptance Criteria:
1. Focus switch updates terminal source within one polling cycle.
2. Poll errors for one lane do not freeze other active lanes.
3. Stream rendering remains responsive under high output.
- Out of Scope: full VT emulation replacement and transcript diff visualization.

### M4.8.6 Per-Lane Intervention Controls

- Labels: `phase-4`, `area-ui`, `area-core`, `type-feature`
- Estimate: `M`
- Dependencies: `M4.8.5`
- Problem: Input/stop actions must be unambiguously lane-scoped when adapter keys repeat.
- Scope:
1. Keep intervention controls bound to selected session ID.
2. Add clear guardrails/errors for invalid lane lifecycle transitions.
3. Preserve auditability of intervention events per session artifact stream.
- Acceptance Criteria:
1. Input sent from lane A never appears in lane B.
2. Stop/interrupt for lane A leaves lane B running.
3. Invalid intervention attempts show clear feedback and do not corrupt state.
- Out of Scope: resume protocol redesign and cross-lane command fan-out.
