# P4.9.8 Multi-Workspace Thread Orchestration Pack

Last updated: 2026-03-10

Purpose: track implementation progress for expanding orchestration so users can create threads from any folder, while creating a new git worktree when launching an additional thread for the same source folder.

## Scope

1. Allow per-thread folder selection (not tied to one global workspace only).
2. Support multiple threads targeting the same source folder.
3. When a same-folder thread already exists, launch the new thread in a dedicated worktree.
4. Preserve existing safety/capability gates and lifecycle cleanup guarantees.

## Current State Snapshot

1. Interactive start path now resolves canonical `source_root` + `repo_root` and supports any selected folder path.
2. Same-folder collision policy is active: if a running thread already targets the same source folder, a dedicated worktree is created for the new thread.
3. Interactive sessions now expose path metadata (`source_root`, `repo_root`, `effective_cwd`, `worktree_path`) in start + list IPC payloads.
4. Orchestration create panel now includes a per-thread folder input with a workspace reset action.
5. Session rail now shows effective execution path and a worktree marker (`wt`) when applicable.
6. Managed worktrees are cleaned up on stop/completion/failure/shutdown, and also on PTY launch failure.
7. Orchestration launch no longer blocks on dirty git worktrees; dirty-state handling remains in race/review flows.
8. Workspace selection is now feature-scoped: race has its own workspace input; orchestration uses per-thread folder input only (no redundant settings workspace field).
