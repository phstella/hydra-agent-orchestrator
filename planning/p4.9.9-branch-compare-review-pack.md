# P4.9.9 Orchestration Branch Compare + Reviewer Analysis Pack

Last updated: 2026-03-10

Purpose: track implementation progress for adding an orchestration-native compare pane that can compare multiple worktree branches of a selected repository and run a reviewer agent against that compare context.

## Scope

1. Add a compare pane in Orchestration.
2. Allow selecting one repository root, one base ref, and multiple target refs.
3. Include refs from local branches and worktree-associated branches.
4. Render compare summaries (per-target change stats and metadata).
5. Launch reviewer analysis from selected CLI tool (`claude`/`codex`) with predefined prompt.
6. Execute reviewer runs at repository root so all compared refs/worktrees are accessible.
7. Render reviewer output in formatted GUI sections with raw-text fallback.

## Constraints / Locked Decisions

1. This is Phase 4 orchestration scope, not Phase 5 workflow preset scope.
2. Tool invocation remains direct CLI wrapping (no feature normalization layer).
3. Reviewer compare prompt is predefined and versioned by Hydra.
4. Reviewer run `cwd` is always repo root (`repo_root`) for compare-analysis launches.
5. Race/review/merge semantics remain unchanged.

## Proposed Runtime Shape

1. Add compare IPC for refs discovery and summary generation:
   - `list_compare_refs(repo_root)`
   - `compare_repo_refs(repo_root, base_ref, target_refs[])`
2. Add compare-review launch IPC:
   - `start_compare_review_session(...)` (or explicit mode on existing interactive start command)
3. Persist compare-review context in session metadata:
   - `repo_root`, `base_ref`, `target_refs`, `prompt_template_version`
4. Add frontend compare pane state machine:
   - repo root selection -> refs load -> compare run -> reviewer launch -> formatted result rendering

## Risks

1. Large multi-target compare payloads can degrade UI responsiveness.
2. Branch/worktree ref discovery can drift across platforms if ref parsing is brittle.
3. Reviewer prompt output may vary; parser must tolerate partial structure.
4. Repo-root launches must not break existing thread/worktree isolation assumptions outside compare-review mode.

## Initial Acceptance Checklist

- [ ] Compare pane supports selecting multiple target refs.
- [ ] Summaries are deterministic and refresh correctly on ref changes.
- [ ] Reviewer launch uses selected adapter and predefined prompt.
- [ ] Reviewer runs from `repo_root` and can inspect all selected refs/worktrees.
- [ ] Formatted output + raw fallback both function.
- [ ] Existing race and review flows are regression-free.
